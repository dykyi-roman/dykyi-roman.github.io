## БЛОК 0 — Общая архитектура приложения

### TASK-001 — Структура и навигация: три брокера

**Приоритет:** 🔴 Высокий — блокирующая задача  
**Зависимости:** Нет  

**Описание:**  
Всё приложение строится вокруг выбора брокера. На верхнем уровне пользователь выбирает брокер, затем внутри него — конкретный паттерн или туториал.

**Что сделать:**

- Добавить верхнюю навигацию из трёх переключателей:
  - `RabbitMQ`
  - `Apache Kafka`
  - `Redis Pub/Sub`
- Активный брокер визуально выделен (цвет, акцент)
- При переключении брокера:
  - Полностью перерисовывается панель режимов (табы второго уровня)
  - Перерисовываются карточки producers и consumers под данный брокер
  - Сбрасываются очередь, счётчики, лог
- Каждый брокер имеет свою цветовую схему акцентов:
  - RabbitMQ — оранжевый (`#FF6600`)
  - Kafka — синий (`#0078D4`)
  - Redis — красный (`#DC382D`)

### TASK-002 — Общие UI-компоненты

**Приоритет:** 🔴 Высокий — блокирующая задача  
**Зависимости:** TASK-001  

**Описание:**  
Набор переиспользуемых UI-элементов и механик, которые используются во всех режимах и брокерах.

**Что сделать:**

- Кнопка "Отправить событие"

- Область для бейджей статуса (ACK, NACK, Reply — появляются ситуативно)
- Правый порт-коннектор (точка соединения с pipe)

**Карточки Consumers:**
- Поле имени сервиса
- Иконка / эмодзи
- Блок подписок / binding keys / паттернов (редактируемый в некоторых режимах)
- Progress bar активности
- Счётчик полученных сообщений
- Левый порт-коннектор (точка соединения с pipe)

**Центральный блок брокера:**
- Название и тип брокера
- Индикатор статуса (live / error)
- Визуализация очереди / топиков / каналов (зависит от брокера)
- Панель режимов (второй уровень навигации)
- Описание текущего паттерна (2–3 строки)
- Мини-график throughput (12 последних секунд)

- Траектория — кривая Безье (кубическая) через JS `requestAnimationFrame`
- Скорость анимации: 400–600ms producer→broker, 400–600ms broker→consumer
- На точке может отображаться текстовый лейбл (routing key, id, ACK и т.д.)
- Одновременно может лететь несколько точек независимо

**Event Log:**
- Фиксированная высота, прокрутка вверх
- Каждая запись: timestamp, тип события (цветовой код), текст
- Типы: `SEND`, `ROUTE`, `RECV`, `ACK`, `NACK`, `ERROR`, `REPLY`
- Кнопка очистки лога
- Лог обновляется при каждом событии в системе

**Общие счётчики (stats bar):**
- Отправлено
- Доставлено
- В очереди
- msg/sec (throughput)

## БЛОК 1 — RabbitMQ Tutorial 1: Hello World

### TASK-004 — Режим Hello World

**Приоритет:** 🔴 Высокий  
**Зависимости:** TASK-002  

**Описание:**  
Простейший паттерн: один producer отправляет сообщение в одну очередь, один consumer читает из неё. Никаких exchange, никакого роутинга.

**Что отобразить в UI:**

**Топология:**
- 1 producer (например, `Sender`)
- 1 очередь в центре (прямоугольник с именем `hello`)
- 1 consumer (например, `Receiver`)

**Поведение:**
- Кнопка "Отправить" на producer отправляет сообщение
- Точка летит: producer → очередь → consumer
- В очереди точка задерживается на ~300ms (визуальное "накопление"), затем летит к consumer
- Если кликать быстро — в очереди накапливается несколько точек, consumer разбирает их по одной
- Очередь визуально показывает количество сообщений внутри (счётчик и/или иконки)

**Описание паттерна в UI:**  
`Самый простой паттерн: P → Queue → C. Producer пишет в именованную очередь, Consumer читает из неё. Один к одному.`

## БЛОК 2 — RabbitMQ Tutorial 2: Work Queues

### TASK-005 — Режим Work Queues (Competing Consumers)

**Приоритет:** 🔴 Высокий  
**Зависимости:** TASK-002  

**Описание:**  
Одна очередь, несколько consumers. Каждое сообщение достаётся ровно одному consumer (round-robin). Паттерн "конкурирующие потребители" — для распределения нагрузки.

**Что отобразить в UI:**

**Топология:**
- 1 producer (`Task Producer`)
- 1 очередь `task_queue` в центре
- 3 consumers (`Worker 1`, `Worker 2`, `Worker 3`)

**Поведение:**
- Сообщения распределяются round-robin: первое → Worker 1, второе → Worker 2, третье → Worker 3, четвёртое → Worker 1 и т.д.
- Указатель round-robin визуализирован: стрелка или подсветка следующего worker
- Каждый worker "обрабатывает" сообщение: progress bar на карточке заполняется за 1–2 секунды
- Пока worker занят обработкой — он не берёт новые сообщения (карточка показывает статус `busy`)
- Если все workers заняты — сообщения накапливаются в очереди
- После завершения обработки worker становится `ready` и берёт следующее сообщение

**Дополнительно:**
- На карточке каждого worker отображается: количество обработанных задач, текущий статус (`ready` / `busy`)

**Описание паттерна в UI:**  
`Work Queue распределяет задачи между workers по round-robin. Каждое сообщение обрабатывается ровно одним worker. Используется для параллельной обработки тяжёлых задач.`

## БЛОК 3 — RabbitMQ Tutorial 3: Publish/Subscribe

### TASK-006 — Режим Publish/Subscribe (Fanout Exchange)

**Приоритет:** 🔴 Высокий  
**Зависимости:** TASK-002  

**Описание:**  
Producer отправляет сообщение в `fanout exchange`. Exchange рассылает копию сообщения во все привязанные очереди. Каждый consumer имеет свою очередь.

**Что отобразить в UI:**

**Топология:**
- 1–2 producers
- Блок `fanout exchange` внутри брокера (визуально отдельный элемент)
- 3–4 очереди (временные, анонимные), каждая привязана к своему consumer
- 3–4 consumers

**Поведение:**
- Точка летит от producer к exchange
- Из exchange одновременно вылетают 3–4 копии (параллельные анимации) к каждой очереди
- Из каждой очереди точка летит к своему consumer
- Все consumers получают каждое сообщение — это ключевое отличие от Work Queues
- Очереди показываются как промежуточный узел между exchange и consumer

**Описание паттерна в UI:**  
`Fanout Exchange рассылает каждое сообщение всем подписчикам одновременно. Используется для широковещательных уведомлений — логирование, кэш-инвалидация, realtime-обновления.`

## БЛОК 4 — RabbitMQ Tutorial 4: Routing

### TASK-007 — Режим Routing (Direct Exchange)

**Приоритет:** 🔴 Высокий  
**Зависимости:** TASK-002  

**Описание:**  
Producer указывает `routing key` при отправке. `Direct exchange` доставляет сообщение только тем очередям, чей `binding key` точно совпадает с routing key.

**Что отобразить в UI:**

**Топология:**
- 2 producers (например, `App Logger` и `DB Logger`)
- Блок `direct exchange` внутри брокера
- 3 очереди с binding keys: `error`, `warning`, `info`
- 3 consumers: `Error Handler`, `Warning Monitor`, `Info Logger`

**Поведение:**
- При нажатии "Отправить" на карточке producer появляется inline-дропдаун с выбором routing key: `error`, `warning`, `info`
- После выбора точка летит к exchange с лейблом выбранного ключа
- Exchange сравнивает routing key с binding keys очередей
- Точка летит только в совпавшую очередь, затем к её consumer
- Очереди/consumers без совпадения кратковременно мигают серым ("отклонено")
- Binding key совпавшей очереди подсвечивается в момент матчинга

**Описание паттерна в UI:**  
`Direct Exchange доставляет сообщение только в очереди с точным совпадением binding key. Routing key задаётся producer при отправке.`

## БЛОК 5 — RabbitMQ Tutorial 5: Topics

### TASK-008 — Режим Topics (Topic Exchange)

**Приоритет:** 🔴 Высокий  
**Зависимости:** TASK-002  

**Описание:**  
`Topic exchange` с паттернами подписок. Routing key — строка слов через точку (`orders.eu.created`). В паттернах: `*` — ровно одно слово, `#` — ноль и более слов.

**Что отобразить в UI:**

**Топология:**
- 2–3 producers с разными примерами routing keys
- Блок `topic exchange` внутри брокера
- 4 consumers с разными паттернами подписок

**Дефолтные паттерны consumers:**

| Consumer | Паттерны подписки |
|---|---|
| Inventory Service | `orders.#`, `payments.#` |
| Email Worker | `*.notifications`, `orders.created` |
| Analytics DB | `#.tracked`, `payments.*` |
| Audit Logger | `#` |

**Поведение:**
- При нажатии "Отправить" открывается inline-инпут для ввода произвольного routing key
- Рядом — кнопки быстрого выбора: `orders.eu.created`, `payments.failed`, `user.login`
- Валидация: только буквы, цифры, точки, `*`, `#`
- После отправки запускается матчинг по всем паттернам всех consumers
- Точки летят параллельно ко всем совпавшим consumers
- Паттерн, который сработал, кратковременно подсвечивается зелёным на карточке consumer
- Consumers без совпадений мигают серым
- Паттерны на карточках consumers редактируемые (клик на тег → инпут)
- Можно добавлять (`+`) и удалять (`×`) паттерны на лету

**Таблица матчинга для тестирования (внутренняя логика):**

| Паттерн | Routing Key | Результат |
|---|---|---|
| `orders.*` | `orders.created` | ✅ |
| `orders.*` | `orders.eu.created` | ❌ |
| `orders.#` | `orders.eu.created` | ✅ |
| `#` | `anything.at.all` | ✅ |
| `*.error` | `payments.error` | ✅ |
| `*.error` | `error` | ❌ |

**Описание паттерна в UI:**  
`Topic Exchange матчит routing key по паттернам: * — ровно одно слово, # — любое количество слов. Позволяет гибко фильтровать события по иерархии.`

## БЛОК 6 — RabbitMQ Tutorial 6: RPC

### TASK-009 — Режим RPC (Request/Reply)

**Приоритет:** 🔴 Высокий  
**Зависимости:** TASK-002  

**Описание:**  
Producer отправляет запрос с `correlation_id` и именем `reply_to` очереди. Consumer обрабатывает запрос и отправляет ответ в reply_to очередь. Producer читает ответ оттуда.

**Что отобразить в UI:**

**Топология:**
- 1 producer (`RPC Client`)
- 1 RPC очередь (`rpc_queue`) в центре
- 1–2 consumers (`RPC Server`)
- Временный узел `reply_queue` — появляется при отправке запроса

**Поведение — фаза запроса:**
- Producer отправляет сообщение, генерируется `correlation_id` (короткий hex, например `a3f2`)
- Под блоком брокера появляется элемент `reply_queue · a3f2` со статусом `waiting...`
- Точка с лейблом `req · a3f2` летит от producer → rpc_queue → consumer

**Поведение — фаза обработки:**
- Карточка consumer пульсирует 800–1200ms (имитация обработки)
- Статус reply_queue меняется на `processing...`

**Поведение — фаза ответа:**
- Точка с лейблом `reply · a3f2` летит от consumer → reply_queue → producer
- Статус reply_queue меняется на `reply received ✓`
- Элемент reply_queue плавно исчезает через 1.5 секунды
- На карточке producer появляется бейдж `↩ reply · a3f2 ✓` на 3 секунды
- Счётчик `Replies: N` на карточке producer инкрементируется

**Одновременные запросы:**
- При быстрой отправке нескольких запросов — несколько reply_queue элементов существуют одновременно, каждый со своим id
- Ответы приходят независимо, каждый матчится по correlation_id

**Описание паттерна в UI:**  
`RPC поверх очереди: client отправляет запрос с correlation_id и reply_to очередью. Server обрабатывает и отвечает. Client ждёт ответ в своей временной очереди.`

## БЛОК 7 — RabbitMQ Tutorial 7: Publisher Confirms

### TASK-010 — Режим Publisher Confirms (ACK / NACK)

**Приоритет:** 🔴 Высокий  
**Зависимости:** TASK-002  

**Описание:**  
После записи сообщения в очередь брокер отправляет producer подтверждение: `ACK` (записано успешно) или `NACK` (ошибка). Producer знает о надёжной доставке только после ACK.

**Что отобразить в UI:**

**Топология:**
- Стандартная: producers → broker → consumers
- Дополнительный визуальный элемент: обратный канал подтверждений

**Поведение — ACK (штатный сценарий):**
- Сообщение летит от producer к брокеру
- Брокер "принимает" (~300ms задержка)
- Маленькая зелёная точка с лейблом `ACK ✓` летит обратно от брокера к producer
- Карточка producer кратковременно подсвечивается зелёным
- Сообщение продолжает лететь к consumers параллельно с ACK
- Счётчик `ACK: N` на карточке producer инкрементируется

**Поведение — NACK (через кнопку Simulate Broker Error):**
- Сообщение долетает до брокера, но не записывается
- Красная точка с лейблом `NACK ✗` летит обратно к producer
- Карточка producer кратковременно подсвечивается красным + CSS shake animation
- Consumers не получают сообщение
- Счётчик `NACK: N` на карточке producer инкрементируется
- В логе: `[CONFIRM] msg_id=N → NACK ✗ · message dropped`
- После NACK режим возвращается к штатному (следующие сообщения снова ACK)

**Глобальный тоггл:**
- В хедере добавить тоггл `Publisher Confirms: OFF / ON`
- Когда `ON` — ACK/NACK механика активна поверх любого другого режима
- В режиме таба `Publisher Confirms` тоггл принудительно `ON` и недоступен для изменения

**Описание паттерна в UI:**  
`Publisher Confirms — брокер подтверждает запись каждого сообщения через ACK/NACK. Гарантия at-least-once delivery на стороне producer.`

## БЛОК 8 — Apache Kafka — Core Concepts

### TASK-011 — Переключение на брокер Kafka: базовая топология

**Приоритет:** 🔴 Высокий  
**Зависимости:** TASK-001  

**Описание:**  
При выборе брокера `Apache Kafka` центральный блок перерисовывается под концепции Kafka: топики с партициями, producer groups, consumer groups с offset tracking.

**Что отобразить в UI:**

**Центральный блок Kafka:**
- Топик состоит из N партиций (по умолчанию 3)
- Каждая партиция — горизонтальная полоса с ячейками (offset 0, 1, 2, 3...)
- Новые сообщения добавляются в конец партиции (append-only лог)
- Каждая ячейка имеет номер offset
- Текущий offset каждого consumer group — отдельный указатель на партиции

**Топология по умолчанию:**
- 2 producers (`Producer A`, `Producer B`)
- 1 топик `events` с 3 партициями (`P0`, `P1`, `P2`)
- 2 consumer groups по 2–3 consumers в каждой

**Настройки (панель в блоке Kafka):**
- Количество партиций: 1 / 2 / 3 / 4 (переключатель)
- Replication factor: 1 / 2 / 3 (визуально — дублирование партиций, лидер и реплики)
- При изменении партиций — топология перерисовывается

---

### TASK-012 — Kafka: распределение по партициям

**Приоритет:** 🔴 Высокий  
**Зависимости:** TASK-011  

**Описание:**  
Визуализация трёх стратегий выбора партиции при отправке сообщения.

**Три режима (переключатель в UI):**

**Round-Robin (без ключа):**
- Сообщения распределяются по партициям по кругу: 0 → 1 → 2 → 0 → ...
- Гарантий порядка нет

**Key-Based Partitioning:**
- Producer задаёт `partition key` (например `user_id=42` или `order_id=7`)
- Партиция определяется: `hash(key) % numPartitions`
- Все сообщения с одинаковым ключом всегда попадают в одну партицию
- В UI: инпут для ввода ключа, рядом показывается `hash → partition N`
- Быстрый выбор: `user_id=1`, `user_id=2`, `region=eu`

**Manual (явное указание партиции):**
- Producer выбирает партицию напрямую из дропдауна при отправке

**Что отображать:**
- Точка летит от producer к конкретной партиции с лейблом ключа
- Ячейка партиции, в которую попало сообщение, подсвечивается
- Offset в ячейке инкрементируется

### TASK-013 — Kafka: Consumer Groups и Offset Tracking

**Приоритет:** 🔴 Высокий  
**Зависимости:** TASK-011  

**Описание:**  
Визуализация ключевых свойств consumer groups в Kafka: каждая партиция обслуживается ровно одним consumer в группе, группы независимы, offset хранится per-group.

**Что отобразить в UI:**

**Consumer Groups:**
- Две группы: `Group A` (3 consumers) и `Group B` (2 consumers)
- Каждая группа — визуально обособленный блок с рамкой и заголовком
- Над каждым consumer — назначенные ему партиции (assignment)

**Offset Tracking:**
- На каждой партиции — два цветных указателя: один для Group A, один для Group B
- Указатель показывает текущий offset чтения группы
- При получении сообщения consumer группы — указатель сдвигается вправо
- Если Group B отстаёт — её указатель визуально позади Group A (lag визуализирован)

**Consumer Lag:**
- Под каждой партицией — индикатор lag: `lag: N msg` для каждой группы
- Если lag > 5 — индикатор становится оранжевым

**Rebalance (кнопка "Trigger Rebalance"):**
- Кнопка в панели управления
- При нажатии: все consumers в группе кратковременно показывают статус `rebalancing...`
- Назначения партиций перераспределяются (анимированно)
- Новые assignment показываются над карточками consumers

### TASK-014 — Kafka: Retention и Replay

**Приоритет:** 🟡 Средний  
**Зависимости:** TASK-013  

**Описание:**  
Ключевое отличие Kafka от брокеров сообщений: сообщения хранятся в логе и доступны для повторного чтения (replay). Визуализация retention и механики `seek`.

**Что отобразить в UI:**

**Retention:**
- Партиция показывает N последних сообщений (по умолчанию 8 ячеек видимо)
- Старые сообщения визуально "выцветают" (уменьшается opacity)
- Кнопка `+ Retention` / `- Retention`: добавляет/убирает видимые ячейки

**Seek / Replay:**
- Кнопка `⏮ Replay from offset` на карточке consumer group
- При нажатии — инпут для ввода offset числа
- Consumer group "перематывает" указатель на указанный offset
- Анимация: указатель двигается влево к нужному offset
- Сообщения начинают читаться повторно от нового offset
- В логе: `[SEEK] Group A · P0 → offset 2`

**Описание паттерна в UI:**  
`Kafka хранит сообщения в упорядоченном логе. Consumer group читает независимо и может перемотать offset назад для replay. Сообщения не удаляются после прочтения.`

---

### TASK-015 — Kafka: Режим Exactly-Once / Idempotent Producer

**Приоритет:** 🟢 Низкий  
**Зависимости:** TASK-012  

**Описание:**  
Визуализация гарантий доставки: at-most-once, at-least-once, exactly-once.

**Три режима (переключатель):**

**At-Most-Once:**
- Producer не ждёт подтверждения (`acks=0`)
- Сообщение улетело — считается отправленным
- Иногда сообщение "теряется": 1 из 5 случайно не долетает (визуально точка исчезает в полёте)

**At-Least-Once:**
- Producer ждёт ACK от лидера (`acks=1`)
- При ошибке — повторная отправка (retries)
- Дублирование: иногда сообщение доставляется дважды (две точки с одним id прилетают)

**Exactly-Once:**
- Idempotent producer: каждое сообщение имеет уникальный `producer_id + sequence_number`
- Дубликаты отбрасываются брокером
- На партиции показывается `seq: N` для каждого сообщения
- Никаких потерь и дублей

### TASK-016 — Переключение на брокер Redis: базовая топология

При выборе брокера `Redis Pub/Sub` центральный блок перерисовывается под концепции Redis: каналы (channels), fire-and-forget семантика, отсутствие персистентности.

**Топология по умолчанию:**
- 3 publishers
- 4 channels
- 4 subscribers

### TASK-017 — Redis: Базовый Pub/Sub

**Приоритет:** 🔴 Высокий  
**Зависимости:** TASK-016  

**Описание:**  
Классический Redis Pub/Sub: publisher отправляет сообщение в канал, все активные подписчики получают его мгновенно. Нет очереди, нет хранения.

**Что отобразить в UI:**

**Поведение:**
- Publisher выбирает канал из дропдауна при отправке
- Точка мгновенно летит к каналу (быстрее чем в RabbitMQ — Redis fire-and-forget)
- Из канала одновременно вылетают копии ко всем активным подписчикам канала
- Подписчики, неактивные в момент отправки (кнопка `Disconnect` на карточке) — не получают ничего
- Визуальный индикатор: "потерянное сообщение" если нет ни одного подписчика (точка летит в канал и исчезает с иконкой 🗑)

**Подписка/отписка в реальном времени:**
- На карточке каждого subscriber — кнопка `Connect / Disconnect`
- При disconnect: subscriber становится серым, порт-коннектор гаснет
- При connect: subscriber возвращается к активному виду
- Изменение отражается мгновенно — не требует перезагрузки

**Описание паттерна в UI:**  
`Redis Pub/Sub — fire-and-forget. Сообщения не хранятся. Если подписчик отключён в момент отправки — он не получит сообщение. Подходит для realtime-событий без гарантий.`

---

### TASK-018 — Redis: Pattern Pub/Sub (PSUBSCRIBE)

**Приоритет:** 🟡 Средний  
**Зависимости:** TASK-017  

**Описание:**  
Redis поддерживает подписку по glob-паттернам через `PSUBSCRIBE`. Подписчик указывает паттерн вида `chat:*` или `events:?:created`, и получает сообщения из всех каналов, соответствующих паттерну.

**Что отобразить в UI:**

**Поведение:**
- Переключатель на карточке subscriber: `SUBSCRIBE` / `PSUBSCRIBE`
- В режиме `PSUBSCRIBE` поле подписки становится инпутом для паттерна
- Поддерживаемые символы: `*` (любое количество символов), `?` (ровно один символ), `[abc]` (один из символов)
- При отправке в канал `chat:room1`: получают все subscribers с паттернами `chat:*`, `*:room1`, `*`
- Сработавший паттерн подсвечивается на карточке subscriber
- В логе: `[PMATCH] events:eu:created → pattern events:*:created matched Subscriber 2`

**Примеры паттернов для быстрого выбора:**
- `chat:*`
- `events:*:created`
- `alerts:?`
- `*`

### TASK-019 — Redis Streams: базовый Event Sourcing паттерн

**Приоритет:** 🟡 Средний  
**Зависимости:** TASK-016  

**Описание:**  
Redis Streams — более мощный примитив по сравнению с Pub/Sub: сообщения хранятся, поддерживается consumer groups и offset tracking. Переключатель `Pub/Sub` / `Streams` внутри раздела Redis.

**Что отобразить в UI:**

**Центральный блок в режиме Streams:**
- Stream как лог сообщений (аналогично Kafka partition, но без партиций)
- Каждое сообщение имеет auto-generated ID вида `1609459200000-0`
- Лог прокручивается слева направо, новые сообщения добавляются справа
- Видимо последние 8–10 сообщений

**Consumer Groups (аналог Kafka):**
- `XREADGROUP` — consumer group читает следующее непрочитанное сообщение
- Два consumer group с независимыми позициями чтения
- Offset-указатели как в TASK-013

**PEL (Pending Entry List):**
- Сообщение считается "pending" пока consumer не подтвердит обработку (`XACK`)
- Pending сообщения подсвечиваются особым цветом на карточках consumers
- Кнопка `XACK` на карточке consumer — подтверждает обработку

**Replay:**
- Кнопка `XREAD from 0` — читать stream с самого начала
- Consumer group перематывается к старту, сообщения читаются заново

**Описание паттерна в UI:**  
`Redis Streams — персистентный лог событий с consumer groups. В отличие от Pub/Sub, сообщения хранятся и доступны для повторного чтения. Поддерживает XACK для at-least-once гарантий.`

---

### TASK-020 — Redis: Event Sourcing паттерн

**Приоритет:** 🟡 Средний  
**Зависимости:** TASK-019  

**Описание:**  
Визуализация паттерна Event Sourcing поверх Redis Streams: состояние системы восстанавливается путём воспроизведения событий из лога.

**Что отобразить в UI:**

**Дополнительный элемент — State View:**
- Справа от потребителей — панель `Current State`
- Показывает агрегированное состояние (например: `orders: 3`, `total_amount: $450`, `last_event: order.shipped`)
- Состояние пересчитывается при каждом новом событии

**Поддерживаемые типы событий (producer выбирает тип):**
- `order.created` — добавляет заказ в state
- `order.paid` — меняет статус заказа
- `order.shipped` — меняет статус
- `order.cancelled` — удаляет из активных

**Кнопка `Rebuild State`:**
- Сбрасывает State View до пустого состояния
- Начинает воспроизводить все события из лога по одному (с задержкой 400ms каждое)
- State View обновляется после каждого события
- Визуализирует саму суть Event Sourcing: состояние = результат применения всех событий

**Описание паттерна в UI:**  
`Event Sourcing: состояние системы не хранится напрямую — оно восстанавливается воспроизведением событий. Redis Stream хранит неизменяемый лог. Rebuild воспроизводит историю заново.`
